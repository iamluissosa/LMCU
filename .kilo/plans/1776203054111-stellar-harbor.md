# Plan de Implementación - QA Report LMCU ERP

## Resumen Ejecutivo

Este plan aborda los hallazgos del informe de auditoría QA & Arquitectura para el sistema LMCU ERP. El sistema es un ERP robusto basado en Next.js 14 y NestJS con arquitectura monorepositorio.

---

## Hallazgos Identificados

| #   | Componente                                                       | Severidad   | Estado        |
| --- | ---------------------------------------------------------------- | ----------- | ------------- |
| 1   | Seguridad Backend: Falta verificación de companyId en PUT/DELETE | **CRÍTICA** | ✅ COMPLETADO |
| 2   | Validación API: Uso excesivo de `any` y DTOs no validados        | **ALTA**    | Por hacer     |
| 3   | Navegación UI: Uso de `window.location.href`                     | **MEDIA**   | Por hacer     |
| 4   | Deuda Técnica: Duplicidad de lógica en roles                     | **MEDIA**   | Por hacer     |
| 5   | Escalabilidad DB: Ausencia de índices                            | **BAJA**    | Por hacer     |

## Fase 2: Estandarización de Capa de Datos (ALTA)

### 2.1 DTOs con class-validator

**Módulos a actualizar** (los que no tienen validación):

| Módulo             | DTOs requeridos                                                       |
| ------------------ | --------------------------------------------------------------------- |
| products           | CreateProductDto, UpdateProductDto                                    |
| suppliers          | CreateSupplierDto, UpdateSupplierDto (ya existen, agregar validación) |
| payments-out       | CreatePaymentsOutDto (agregar validaciones)                           |
| incomes            | CreateIncomeDto (agregar validaciones)                                |
| departments        | CreateDepartmentDto, UpdateDepartmentDto                              |
| expense-categories | CreateExpenseCategoryDto, UpdateExpenseCategoryDto                    |

### 2.2 Eliminación de `any` Types

**Archivos con mayor incidencia** (ordenar por prioridad):

1. `sales/sales-invoices.service.ts` - 10+ instancias
2. `sales/quotes.service.ts` - 8+ instancias
3. `payments-out/payments-out.service.ts` - 8+ instancias
4. `suppliers/suppliers.service.ts` - 3+ instancias (fácil)
5. `dashboard/dashboard.service.ts` - 6+ instancias

**Estrategia**:

- Reemplazar `data: any` con tipos específicos
- Eliminar casts `as any` y `as unknown`
- Usar los DTOs existentes o crear nuevos
